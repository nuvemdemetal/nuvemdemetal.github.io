Subindo cluster k8s com Ansible, Terraform e Shell Script

Tive uma experiência ruim ao usar uma conta compartilhada da AWS para fazer laboratórios k8s. Mesmo atendendo a todos os padrões de TAGS e tudo mais insistiam em apagar meu Cluster. Eu decidi então direcionar a raiva para uma coisa boa e com a alta do dólar foi realmente totalmente excelente subir e destruir meu cluster a hora que eu quiser em minha própria conta.

Entre os colegas mais pŕoximos não encontrei nada 100% automatizado. Uma coisa ou outra. E trabalhei alguns dias em algo pra mim e para a comunidade. Então o que tenho aqui é um Shell Script que orquestra todo o processo de construção de um cluster k8s. Desde o provisionamento da infraestrutura com Terraform, instalação dos pacotes com Ansible e manipulação dos IP’s com Shell Script para instalação correta dos pacotes de acordo com o tipo de nó.

Desconstruindo o Frankstein

Faça algumas alterações:

  1. Dentro do Script MONTE_O_CLUSTER.sh, depois da fase de alteração do SG, coloque o caminho do diretório do Ansible local da sua máquina.
  2. Use sua access_key e secret_key no main.tf, diretório terraform e também no ec2.ini para que o ec2.py possa fazer a varredura adequadamente em sua conta AWS e encontrar as instâncias TAGUEADAS.
  3. Conhecendo o conceito de chave pública, adicione a sua em public_key no main.tf .

Fluxo:

Dividi em 3 fases os gatilhos do Script principal, MONTE_O_CLUSTER.sh :

FASE 1: O Terraform subirá 3 instâncias na AWS

FASE 2: Scripts em Shell manipulam informações junto ao ec2.py (inventário dinâmico) na conta AWS para obter quais IP’s receberam diferentes tipos de TAGS e anexam isso ao Terraform, que aplica as mudanças na Infraestrutura AWS.

FASE 3: O Ansible é acionado, e faz as instalações em cada nó de acordo com a TAG.

TERRAFORM

Com o Terraform devidamente funcional, preparei o main.tf para construir os SG’s tão importantes no Cluster, e também definir o tipo de instância que serão usadas, e com isso setar o SG correto para as instâncias, no caso usei um Cluster com 3 instâncias, sendo um Master e dois Workers (que chamei de Puppets).

O objetivo no primeiro momento foi fazer funcionar, então se o seu objetivo é ter todas as boas práticas possíveis fique a vontade para contribuir com o código. Lembrando que a liberação das portas fica ao seu critério.

Mas porque tantos SG’s ??? >

Existem regras no SG liberando o tráfego entre as instâncias, que em primeiro momento foram descritas como worker1, worker2 e master1 com todas as portas necessárias para que os componentes do cluster possam funcionar, e além disso, caso queira também instalar o Istio, as portas já estarão liberadas. Então ./SEPARA_IP_DO_MASTER.sh :

#!/bin/bash

./ec2.py --list > tags.json

worker=`cat tags.json | jq '.tag_type_worker' | grep '[0-9]' | sed 's/\"/ / ;s/\"/ / ;s/\,/ / ' | cut -c 4-`
master=`cat tags.json | jq '.tag_type_master' | grep '[0-9]' | sed 's/\"/ / ;s/\"/ / ;s/\,/ / ' | cut -c 4-  | xargs`

echo -e "[tag_type_worker]\n$worker\n[tag_type_master]\n$master\n" > "adicione aqui, sem as aspas, o diretório ao qual se encontra o script principal > MONTE_O_CLUSTER.sh"

Em seguida, uso o ec2.py para varrer a conta na AWS e identificar quais instâncias receberam determinadas TAGS, que serão escritas em um arquivo chamado tags.json e em seguida enviadas para hosts.json:

[tag_type_worker]
5.55.555.555  
6.666.666.666 
[tag_type_master]
77.777.777.777

Em seguida ./ENCAIXA_IP_NO_TERRAFORM.sh :

#worker1=`sed -n '2p' hosts.json | cut -c -12`
worker1=`sed -n '2p' hosts.json | grep '[0-9]' | xargs`
worker2=`sed -n '3p' hosts.json | grep '[0-9]' | xargs`
master1=`sed -n '5p' hosts.json | grep '[0-9]' | xargs`

sed -i "s/worker1/$worker1/g" main.tf

sed -i "s/worker2/$worker2/g" main.tf

sed -i "s/master1/$master1/g" main.tf

sed -i 's/\#/ /g' main.tf

Se quiser, verifique no seu console AWS as TAGS de cada instância, acesse, e verifique se de fato há conexão entre elas.

O Script MONTE_O_CLUSTER.sh irá fazer o terraform aplicar as mudanças no main.tf, para que em seguida o Ansible identifique através das TAGS qual é a instalação correta para cada tipo de nó do Cluster.

ANSIBLE

Então com a infra provisionada e devidamente TAGUEADA, o Ansible faz muito bem sua parte, dando a César o que é de César, ou seja, instalando os pacotes de Docker e k8s nos três nós, mas fazendo do MASTER o senhor dos PUPPETS!

Playbook site.yml:

- name: Install k8s and docker packages
  become: yes
  hosts: tag_name_k8s
  roles:
    - docker
    - k8s-all

- name: Setup master node
  become: yes
  hosts: tag_type_master
  roles:
    - k8s-master

- name: Setup worker node
  become: yes
  hosts: tag_type_worker
  roles:
    - k8s-worker

Role k8s-master:

- name: Destroy cluster before starting again
  command: kubeadm reset -f

- name: Images Pull
  command: kubeadm config images pull

- name: Initialize k8s cluster
  command: kubeadm init
  register: kubeadm_init_output

- name: Create file with output
  copy:
    content: "{{ kubeadm_init_output.stdout }}"
    dest: /root/kubeadminit

- name: Prepare kubejoin script
  shell: tail -2 /root/kubeadminit > /root/kubejoin.sh

- name: Get kubejoin
  fetch:
    dest: ./join
    src: /root/kubejoin.sh

- name: Organize files
  shell: |
        cp join/{{ inventory_hostname }}/root/kubejoin.sh join/
        rm -fr join/{{ inventory_hostname }}
  delegate_to: localhost
  become: no

- name: Install pod network Calico
  command: kubectl apply -f "https://docs.projectcalico.org/v3.14/manifests/calico.yaml"
  environment:
    KUBECONFIG: /etc/kubernetes/admin.conf

O Nó Master recebe o pod network Cálico, mas recomendo tambem o pod network Weave. Os dois são excelentes e trabalham de formas semelhantes.

E o próprio Ansible constrói o Script kubejoin.sh para posteriormente adicionar os nós ao Cluster.
