Terraform Cloud

O Terraform Cloud é uma ferramenta nativa da Hashicorp usada para facilitar o CI/CD de ambientes com o Terraform.

Benefícios:

  1. Não se preocupe com instalação ou plugins, o Terraform Cloud se encarrega de apenas rodar seu código.
  2. É possível avançar na criação e gerencia dos ambientes de forma auditável com o VCS.
  3. Através do VCS (Uso Github) integrado é possível manter controle de versão do código.
  4. Oferece segurança para dados sensíveis como o tstate.tf, credenciais dos Cloud Providers e isenta uso de tabela no Mongo para Lock.

Não se preocupe com instalação ou plugins, o Terraform Cloud se encarrega de apenas rodar seu código.

Não é necessário fazer a instalação do Terraform localmente. Após fazer a integração do módulo Terraform de dentro do VCS na configuração da Workspace dentro do Terraform Cloud, é preciso rodar o código para subir a Infra apenas na primeira vez. É só clicar em dois botões e pronto, a ferramenta se encarrega das instalações necessárias de acordo com sua versão do Terraform para subir a Infra mostrando todo o log dentro da ferramenta. Para gerenciar a Infra criada à partir da segunda mudança o Terraform Cloud identifica que foi realizada mudança na Branch ou diretório dentro do repositório e fica em fase de “plan” aguardando aprovação do usuário para subir a mudança na Infra:

É possível avançar na criação e gerencia dos ambientes de forma auditável com o VCS e Terraform Cloud.

Como em todo projeto, é de boa prática padronizar um Workflow no VCS para fazer a gerência do código. Costumo usar o gitflow, trancar a branch Main a nível de administrador (evita qualquer acidente ainda que o Terraform Cloud fique em estado de plan após alteração na Branch Main do repo) e exigir o review de ao menos um membro do time + review do dono do código. Branches para trabalhar: (Main e Develop). Sempre que surgir alguma alteração a ser feita na Workspace é de boa prática criar uma nova Branch e deletar após o Merge, mas se você usa o gitflow sabe que ele faz isso por você ;)

Sugiro também padronizar templates para Issues e Pull Requests. Isso evita longas discussões em torno do código.

Através do VCS (Uso Github) integrado é possível manter controle de versão do código.

O controle de versão (feito na release) é importante para referenciar no módulo qual versão estará usando e assim fazer o reaproveitamento do código e poder voltar em determinados pontos da infraestrutura. O Terraform Cloud também oferece as versões do estado da Infra (tfstate.tf) a cada deploy feito, o que é muito bom também para realizar o troubleshooting e poder voltar ao estado da Infra caso alguma merda aconteça ;)

Oferece segurança para dados sensíveis como o tstate.tf, credenciais dos Cloud Providers e isenta uso de tabela no Mongo para Lock.

O tfstate.tf fica armazenado dentro de cada Workspace e existe um controle de versão deste tfstate.tf, algo que não acontece caso seja guardado remotamente (em um bucket S3 por exemplo) pois ele é reescrito todas as vezes. Além disso, como o código roda um por vez dentro da Workspace, não é preciso usar o Lock que é uma ação recomendada a usar quando o tfstate é guardado remotamente.
